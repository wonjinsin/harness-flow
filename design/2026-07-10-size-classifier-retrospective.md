# 크기 분류기(Size Classifier) 작업 회고 — small 티어의 도입과 폐기

**날짜**: 2026-07-10
**대상 브랜치**: `worktree-size-classifier`
**결론 요약**: 3-티어(trivial/small/standard) 분류기를 구현·적대검증·A/B 평가한 결과, **trivial 티어는 채택**(승인 왕복 −81%, 품질 손실 0, 미끼 방어 완봉), **small 티어는 폐기**(과잉 상향으로 절감 효과 소멸). 최종 형태는 trivial/standard 2-티어이며 3차 재평가에서 전 게이트 조건을 충족해 확정(§4). 이 문서는 small을 재시도할 때 반드시 먼저 읽어야 할 기록이다.

## 1. 배경 — 왜 분류기를 만들었나

6개 서드파티 하네스(ECC/GSD/gstack/OMC/Archon/superpowers) 비교 분석에서, harness-flow만 **작업 크기에 비례하는 세리머니 스케일링이 없었다**. 한 줄 변경도 브레인스토밍→스펙 문서→완성 코드 포함 플랜→SDD→다중 리뷰의 풀 체인을 강제 — 사용자 승인 왕복 6~8회, 문서 2개, 구현 사실상 2회. 직전 작업(46386f0, Task Group 코어스닝)이 실행 단계의 토큰/왕복은 −58%/−67% 줄였지만 wall-clock은 오히려 +22~63% 늘었음이 실측됐고(그룹화는 비용 레버지 속도 레버가 아님), 남은 체감 지연의 주범은 **앞단 고정 세리머니와 사용자 대기**로 진단됐다.

## 2. 무엇을 만들었나

- **선판정 + 사후검증 하이브리드**: 부트스트랩(`using-harness-flow`)에 3신호 판정표(변경 규모 / 신규 의존성·계약 / 설계 모호성, 최고 신호 승리) + 상향 트리거(보안·계약·의존성·마이그레이션·동시성·모호성) + "애매하면 상위" 규칙. 출구에서 diff 실측 캡(사후검증)으로 선판정 실패를 회수.
- **A-lite 배치**: 결정-완결 판정표는 상시 주입층(부트스트랩)에, 미끼 예시·캡 수치·소급 절차는 온디맨드 `references/sizing.md`에. 근거: 라우팅은 디스패처의 일이고(버그→systematic-debugging 분기 선례), 상시층에는 결정 로직만.
- **티어별 경로**: trivial = 인라인 TDD + 셀프리뷰 + 커밋 전 누적 diff 캡 (워크트리·문서·승인·디스패치 0) / small = 워크트리 + 채팅 설계 요약 승인 1회 + 인라인 TDD + 최종 리뷰 (스펙/플랜 문서 없음) / standard = 기존 풀 체인 무변경.
- **적대적 red-team 2종** (악용 시나리오 / 모순·우회로) → Critical 3, Important 9 발견, 전부 수정. 핵심 수정: 트리거를 캡 검사 시점에 실제 diff 대상으로 재검증, trivial 누적 캡(2번째 trivial 커밋 = 재분류), trivial 선언 전 sizing.md 필독, small 경로를 brainstorming 경유로 정규화, finishing의 `main` 하드코딩 버그(이 리포는 master — 백스톱 무음 무력화).

## 3. 평가 방법 (재사용 가능)

- **A/B dry-run 시뮬레이션**: 시나리오 10개 × 2 arm(OLD=1.1.3 스킬셋 / NEW=분류기 체인). arm은 sonnet 신선 컨텍스트에서 해당 스킬 파일 전체를 읽고 가상 Node.js 리포에 대해 스킬이 강제하는 플로우를 문서 작성까지 충실히 수행, 승인/디스패치/문서/안전장치를 구조화 트레이스로 반환.
- **블라인드 판정단**: opus, X/Y 익명 트레이스 + arm에게 숨긴 정답 키(정답 티어, 최소 안전장치)로 프로세스 안전성만 판정.
- **시나리오 설계**: trivial 3 / small 3 / standard 컨트롤 1 / **미끼 3** (한 줄처럼 보이는 공개 API 계약 변경, 인증 경로 수정, 설계 선택이 숨은 "금방 하지?" 스크립트).
- **합격선(폐기 게이트)**: trivial/small 토큰 −40%↑ AND 승인 왕복 −50%↑ AND 품질 동등·미끼 누출 0 AND standard 무변화. 하나라도 미달 → 폐기.
- **교훈(측정)**: dry-run 토큰은 프록시로 부적합 — N=1 분산과 추론 장황함이 경로 비용 신호를 삼킴 (같은 lean 경로인데 +16%, 같은 풀 체인인데 −40% 같은 역전 발생). 신뢰 지표는 트레이스에서 결정적으로 세어지는 **승인 왕복·문서 줄수·디스패치 수**. wall-clock 직접 주장 금지, 승인 왕복을 대리지표로 명시할 것.

## 4. 결과

### 1차 (하드닝 직후)
NEW가 6개 정상 시나리오 중 5개를 standard로 과잉 상향. 원인: red-team이 요구한 트리거 확장("계약 추가·변경·삭제")이 사실상 모든 코드 변경에 걸림. 미끼 방어는 3/3.

### 트리거 보정 (1회)
"소비자가 관찰 가능한 기존 계약의 변경/삭제"로 축소, 하위호환 추가는 비트리거, 모호성 트리거는 "요청이 열어둔 선택지"로 한정, 상향에도 트리거 명명 요구(상향 tier-shopping 카운터앵커).

### 2차 (보정 후)
| 지표 | trivial군 | small군 |
|---|---|---|
| 승인 왕복 | **−81%** (T2는 0회, 문서 0줄) | −14% (기준 −50% 미달) |
| 분류 정확도 | 2/3 (T3는 small로 1단 상향) | 1/3 (S2·S3가 standard로 상향) |
| 미끼 방어 | — | **3/3** + 판정단이 OLD를 더 위험 판정 2건 (B2 보안 리뷰 누락, T1 호출부 감사 누락 — NEW의 상향이 실제 결함을 잡음) |

결론: **품질·안전 조건 완승, 효율 조건은 small에서 미달** → 게이트 규칙대로 small 폐기, trivial만 채택 (2-티어 축소, 커밋 c3a1c37).

### 3차 (2-티어 축소 후, 최종)

시나리오 8개: trivial 2(T2, T3) / 미끼 5(T1 내부 계약 파괴 · S1 3파일 스코프 · B1 공개 API · B2 보안 · B3 모호성 — 전부 정답 standard) / 컨트롤 1(C1).

| 조건 | 결과 | 판정 |
|---|---|---|
| trivial군 승인 −50%↑ | 평균 5.5 → 1.0 (**−81.8%**), 문서 165/300줄 → 0줄, 디스패치 0 | 충족 |
| 미끼 누출 0 | 5/5 전부 standard 분류 또는 에스컬레이션 | 충족 |
| standard(C1) 무변화 | 풀 체인 4대 안전장치(스펙/플랜/그룹 리뷰/최종 리뷰) 전부 실행 | 충족 |
| 품질 동등 | quality_flags 0. 판정단 riskier: T1에서 **OLD가 더 위험**(호출부 감사 누락 — NEW는 계약 트리거를 명시 적중해 소비자 영향을 표면화), 나머지 전부 none | 충족 |

**게이트 전 조건 충족 → trivial/standard 2-티어 채택 확정** (release 1.1.4).

## 5. 왜 small은 실패했나 — 구조적 원인

**실패한 것은 "중간 티어" 자체가 아니라 "LLM이 중간 티어를 선판정하는 방식"이다.**

| 하네스 | 중간 티어 작동 방식 | 판정 주체 |
|---|---|---|
| GSD | `/gsd-quick` — 사용자가 커맨드로 선택, LLM은 자격 검증만 (미달 시 거부·정지) | 사용자 |
| ECC | small = "1파일/1함수" — 기계적 경계 | LLM이지만 셀 수 있는 신호만 |
| gstack | 완성된 diff 크기(50-200줄)로 **뒷단 리뷰만** 스케일 — 세리머니가 뒤에 있어 예측 불필요 | 측정값 |
| Archon | 게이트 0~4개 워크플로를 사용자/라우터가 선택 | 사용자 |

harness-flow의 small은 "≤5파일 + 남은 설계 모호성 없음"이라는 **판단형 경계**를 LLM 선판정에 맡겼고, 여기에 "애매하면 상위" 안전규칙이 얹히면 구조적으로 상향 쏠림이 발생한다. 실측 증거: 기계적으로 판정 가능한 S1(옵션 추가+호출부 2곳)은 통과(승인 9→3), 판단이 필요한 S2(로그 로테이션 — sync/async, 체크 주기)·S3(주차 계산 교체 — 반환 형식, 타임존)만 상향.

**정밀도/재현율이 같은 다이얼에 묶여 있다**: S2 요청과 B3 미끼("CSV 임포트, 큰 파일도, 금방 하지?")는 문형이 동형이고 차이는 숨은 선택지의 치명도뿐. 모호성 트리거를 S2가 통과하게 무디면 B3가 샌다. 반복 1회로 1차 5/6 → 2차 3/6 오상향까지 왔지만, 그 이후는 미끼 방어율과 교환하는 구간.

**앞단 세리머니의 근본 제약**: harness-flow의 비싼 세리머니(설계 대화·스펙·플랜)는 코드 존재 *전*에 온다. diff가 없으니 측정 불가 → 예측이 불가피 → 예측 편향 문제를 피할 수 없다. gstack이 사후 실측으로 문제 자체를 안 가지는 것은 세리머니가 뒷단(리뷰)이기 때문. 사후 실측이 앞단 하네스에서 할 수 있는 역할은 백스톱(건너뛴 검증의 소급 복원)까지다.

## 6. small 재시도 시 검토할 방향 (우선순위 순)

1. **사용자 opt-in 전용 (GSD 방식, 최유력)**: 자동 판정 제거. 사용자가 명시 요청("간단하게", "/quick")할 때만 small 경로. LLM은 자격 검증만 — 상향 트리거에 걸리면 small 요청을 거부하고 standard로 (비대칭: 하향은 사용자 권한, 상향 강제는 스킬 권한). 과잉 상향 문제가 정의상 소멸. 추가 평가 필요분: "사용자가 small을 요청한 미끼" 시나리오.
2. **기계적 경계로 재정의 (ECC 방식)**: small = "1파일/1함수 + 트리거 무해당" 같은 셀 수 있는 조건만. 판단형 신호(모호성) 배제. 경계가 좁아져 절감 폭도 줄지만 안정적.
3. **비추천 — 판단형 경계 재튜닝**: 2차 반복에서 한계 확인. 미끼 벽과 트레이드오프.

## 7. 그 외 보존할 교훈

- **red-team 하드닝은 반대편 실패를 만든다**: 낙관 분류를 막는 문구(트리거 확장, 애매하면 상위)가 체계적 비관 분류를 유발했다. 방어 문구를 넣을 때는 반드시 반대 방향 카운터앵커(상향에도 근거 명명 요구)와 함께, 그리고 eval로 양방향을 재라.
- **정답 키도 오류가 있다**: T1(공유 유틸의 동작 파괴 변경)을 trivial로 키잉한 것은 설계 실수였고, 판정단이 이를 드러냈다(OLD가 trivial 취급해 실제 결함 출하). 키 수정은 결과 튜닝이 아니라 키 결함 수정으로만, 사유와 함께 기록.
- **finishing `main` 하드코딩**: 이 리포 기본 브랜치는 master. `git merge-base HEAD main 2>/dev/null || git merge-base HEAD master` 패턴 필수 (finishing Step 3에 선례).
- **워크플로 args**: Workflow 툴 args가 문자열로 도착할 수 있음 — 스크립트에 `typeof args === 'string' ? JSON.parse(args) : args` 가드.
- **워크플로 캐시**: 같은 runId 재개 시 (prompt, opts) 동일하면 캐시 재생 — 스킬 파일 내용이 바뀌어도 프롬프트(경로)가 같으면 낡은 결과가 재생된다. 파일 내용이 바뀐 arm은 라벨(opts)에 반복 번호를 넣어 캐시를 무효화할 것.
